02

The five hostable components

Each card is one deployable unit with its own host profile. The two model services are stateless GPU workloads that can be hosted, scaled, and versioned independently of the application code.

Component 1 · Model serving (GPU)

Transaction Tagging Encoder

Customer-trained encoder (transformer classifier) that reads each transaction's narration + amount + direction and assigns an L2 → L1 category. Turns raw rows into tagged transactions.
Compute
GPU inference (small/mid transformer). CPU-servable at low volume.
State
Stateless. Model weights are the only artifact; versioned per training run.
Scales by
Transaction volume (rows/sec). Batch-friendly, horizontally replicable.
Interface
Batch/REST: rows[] → tags[]. No business logic inside.
Why separate
Retrained on its own cadence; needs GPU; owned by an ML team, not app devs.
Component 2 · Model serving (GPU)

Decoder LLM (Narrative Summaries)

Generative decoder model that turns the structured feature set into an analyst-grade executive review. Prose only — it never computes numbers.
Compute
GPU. Served via Ollama / vLLM (OpenAI-compatible).
Model
Configurable in config/settings.py — default llama3.2; deepseek-r1 supported (think-trace stripping).
State
Stateless per request. Deterministic (temperature=0, seed=42).
Scales by
Concurrent reports. Bottleneck = GPU throughput; queue or replicate.
Interface
REST prompt/response. Fail-soft: runner still emits a report if this is down.
Component 3 · Application runner (CPU)

Agent Runner + Feature Tools

The whole project codebase. An agent that orchestrates deterministic feature-creation tools, assembles the typed CustomerReport, calls the decoder for narration, and renders output.
Compute
CPU. Pure Python (pandas/pydantic). No GPU.
Feature tools
salary detectionEMIscashflow eventsaccount qualitymerchants recurring rulesscorecard
State
Stateless per report; in-process cache only. Config = single source of truth.
Scales by
Reports/hour. Embarrassingly parallel — run N workers behind a queue.
Depends on
Data Store (read) + both model services (network calls).
Component 4 · Data store

Transaction Data Store

System of record for input transactions and auxiliary signals the runner reads (salary-string outputs, category registry reference).
Today
Tab-separated CSVs (rgs.csv, rg_sal_strings.csv) on local disk.
Target
Postgres / BigQuery. Swap is isolated to data/loader.py.
Access
Read-heavy from runner. One admin write path (category overrides).
Scales by
Customers × transaction history. Independent of compute tiers.
Why separate
Data lifecycle, retention, and governance differ from application code.
Component 5 · Output / serving

Report Serving & Artifacts

Persists and serves the generated HTML + Excel reports, and exposes the admin /save endpoint for category overrides that feed corrections back to the data store.
Compute
Lightweight HTTP server (CPU). Can be static hosting + a small API.
Artifacts
Rendered *_report_v2.html and .xlsx per customer.
State
Object storage for artifacts; write-back edits route to Data Store.
Scales by
Concurrent viewers. Fully decoupled from generation.
Why separate
Viewing/serving has a different uptime + auth profile than batch generation.

04

Why this split

🧠

Different hardware

The two models want GPU; the runner and serving want cheap CPU. Splitting them stops you from paying for GPU on the 90% of work that is deterministic Python.

🔁

Different lifecycles

The tagging encoder is retrained; the decoder is swapped by config; the runner ships on app cadence; data has its own retention. Independent deploys mean independent release trains.

📈

Independent scaling

Scale report throughput by adding CPU runners; scale narration by adding GPU decoder replicas; scale tagging by batch size — each bottleneck is addressed in isolation.

🛡️

Blast-radius control

Fail-soft boundaries: if the decoder is down, reports still generate with deterministic content. Model outages degrade quality, they don't stop the pipeline.

👥

Team ownership

ML team owns the two model services; platform team owns the runner and data; each contract above is the hand-off point, so nobody edits across boundaries.

🔌

Swappable parts

Encoder, decoder model, and data backend are each replaceable behind a stable interface — a one-line config change or a single adapter file, not a rewrite.

05

Two ways to host the same five components

The decomposition is the same in dev and prod — only the placement changes.

Scenario A · Single box (dev / demo)

All five co-located

  • Runner, encoder, decoder (local Ollama), CSV data, and the report server all on one machine.
  • Zero network hops; simplest to run (python main.py --customer <id> --serve).
  • Limits: one GPU shared by both models; no horizontal scaling; CSV in memory.
Scenario B · Distributed (production)

Split by hardware & scaling need

  • GPU tier: Tagging Encoder + Decoder LLM as separate model-serving endpoints.
  • CPU tier: N Agent-Runner workers behind a queue/load balancer.
  • Data tier: Postgres/BigQuery replacing CSVs (loader swap only).
  • Serving tier: Object storage + web server for artifacts and admin edits.
Fidelity note (target vs. today). This is the target production view. In the current repo, transaction categories are pre-computed and shipped in the data (category_of_txn / category_of_txn_l2 in rgs.csv) with deterministic keyword rules — there is no hosted encoder model yet. Component 1 above is where that tagging becomes a customer-trained encoder service. Likewise the LLM runs on a local Ollama today; the diagram shows it as a hostable GPU endpoint. Everything else (runner, feature tools, data, rendering/serving) matches the code as-is.
