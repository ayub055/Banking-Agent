Design Principles

Architecture Decisions

This system was built for a lending environment where decisions must be auditable, consistent, and reliable. A handful of core ideas shape how every component is designed — here's what they mean in plain terms.

🎯
AI explains — Python decides
Every risk threshold, DPD flag, and utilization signal comes from hardcoded Python rules in config/thresholds.py. The AI only writes the summary in natural language. It never makes a risk call independently. Same inputs always produce the same findings.
Deterministic Core
🔒
Customer data never leaves the building
All AI models run locally via Ollama — mistral for parsing, llama3.2 for summaries. No transaction or bureau data is sent to OpenAI, Google, or any cloud service. Zero API costs, zero data egress risk.
Data Privacy
🛡️
Reports always complete, even when AI fails
Every LLM call is wrapped in try/except. If a model times out or crashes, the report still generates with all data sections — only the AI-written summary is omitted. No single failure can block report delivery.
Fail-Soft
⚡
Repeat queries return instantly
Transaction data is loaded once at process start and cached. Reports are cached by customer + period. Re-running a bureau report for the same customer costs zero computation and zero AI calls — the cached result is returned immediately.
Multi-Layer Cache
⚖️
Every customer analyzed the same way
The same 40+ threshold rules apply to every customer with no variation. DPD_HIGH_RISK = 90 days, CC_UTIL_HIGH_RISK = 75% — these are constants, not analyst judgement. Results are reproducible: re-running a report produces identical findings.
Consistency
🔧
Add rules without touching pipeline code
Categories (food, rent, betting) live in categories.yaml. Risk thresholds live in thresholds.py. All 7 prompts live in prompts.py. A non-engineer can extend detection coverage by editing config files — no pipeline changes needed.
Configuration-Driven

1Local LLM via Ollama — no cloud API ▼
Banking transaction data and bureau tradelines contain highly sensitive financial PII. Running models locally via Ollama ensures no customer data leaves the organisation's infrastructure. All inference happens on-premises — no token-level billing, no network latency variability, no vendor dependency.
⚠ Trade-off: Lower peak model quality than GPT-4. Mitigated by choosing deepseek-r1:14b for complex report summaries — a reasoning model that approaches frontier quality on structured analytical tasks.
2Three separate models for three different jobs ▼
mistral (parser): Fast, JSON-reliable, low latency. Intent parsing runs on every query — speed matters more than reasoning depth. JSON mode guarantees parseable output.

llama3.2 (explainer): Real-time streaming. User is watching — first-token latency matters most. 3B parameters, no reasoning blocks.

llama3.2 / deepseek-r1:14b (summaries): Report generation is async, quality is critical. deepseek-r1 is the upgrade path for richer cross-signal interpretations in bureau summaries.
3Deterministic core, LLM periphery ▼
Every piece of logic that can be expressed as Python code is expressed as Python code. LLMs are used only for natural language understanding (intent parsing) and prose generation (summaries, explanations). This makes the system:
  • Auditable — deterministic outputs from named threshold constants
  • Testable — pure functions can be unit tested with assert statements
  • Debuggable — failures are in Python stack traces, not LLM black boxes
  • Cheaper — fewer LLM calls per query
4Pydantic v2 schemas at every module boundary ▼
Every interface between pipeline stages is typed: ParsedIntent, ToolResult, TransactionInsights, CustomerReport, BureauReport. Pydantic validates data at runtime and gives IDE type inference across the codebase.

The model_copy(update={...}) pattern (Pydantic v2) is used for immutable report updates — sections from one report cannot bleed into another. This prevents a class of subtle bugs where cached report data is accidentally mutated.
5YAML-driven category configuration ▼
Categories (salary, betting, rent, food, EMI) are defined in config/categories.yaml, not in code. Each entry specifies keywords, aliases, transaction direction, and display name. This allows a non-engineer (product manager, compliance analyst) to add a new detection category or tune keywords without a code deployment.

The @lru_cache on the YAML loader means the file is parsed once per process. resolve_category_alias() handles fuzzy user input (e.g. "food" → "Food & Dining").
6Fail-soft report generation ▼
Every optional section — LLM summaries, tradeline features, detected events — is wrapped in try/except. Reports are generated with whatever data is available. A single LLM timeout does not fail the entire report. If generate_customer_review() raises an exception, the report section is omitted and the rest is delivered.

Design trade-off: A partial report delivered immediately is more valuable than a perfect report that sometimes fails entirely. Operators can see from the report structure which sections were missing and re-run if needed.
7Two-paragraph LLM report structure ▼
All LLM-generated report sections follow a fixed two-paragraph structure:

Paragraph 1 — Financial Overview: LLM synthesises structured numbers (salary, EMI, utilisation) into flowing prose. The numbers are provided as pre-computed facts; the LLM's job is narration, not calculation.

Paragraph 2 — Transaction Events / Behavioral Insights: LLM narrates specific detected events. The prompt mandates inclusion of all events passed to it — this prevents the LLM from "forgetting" a high-risk event that doesn't fit its narrative structure neatly.
8Multi-layer caching with explicit invalidation ▼
Three caching layers, each with a clear invalidation path:
  • DataFrame cache (_transactions_df) — module-level, loaded once. Cleared via load_transactions(force_reload=True).
  • Report cache (_REPORT_CACHE) — keyed by (customer_id, period). Cleared via invalidate_customer_cache().
  • Insight cache (_INSIGHT_CACHE) — keyed by (customer_id, scope). Cleared via clear_all_cache().
No automatic TTL — appropriate for a batch/session analytics tool where data is refreshed daily. Real-time systems would need TTL or event-driven invalidation.
